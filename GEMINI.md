# 절대엄금 기본수칙 (Foundational Mandates)

이 문서는 본 프로젝트의 무결성과 일관성을 유지하기 위한 최상위 지침입니다. 모든 프롬프트 수행 시 이 수칙을 최우선으로 읽고 준수해야 합니다.

## 1. 기존 디자인 및 레이아웃 절대 보존
- 1-1: 현재 적용된 폰트, 폰트 사이즈, 컬러 시스템을 임의로 변경하지 않는다.
- 1-2: 카드 형태의 레이아웃, 여백(Margin/Padding), 그리드 시스템 등 기존 문서의 구조적 설계를 유지한다.
- 1-3: 사이트의 전반적인 분위기와 시각적 질서를 손상시키는 모든 시도를 금지한다.

## 2. UI/UX 일관성 유지
- 2-1: 새로운 기능 추가 시, 기존 사이트의 분위기와 톤앤매너(Look & Feel)를 완벽하게 계승한다.
- 2-2: 사용자 경험에 혼란을 줄 수 있는 이질적인 UI 컴포넌트 사용을 금지한다.

## 3. 반응형 디자인 파괴 금지
- 3-1: 모바일, 태블릿, 데스크탑 등 모든 해상도에서 기존의 최적화된 뷰를 손상시키지 않는다.
- 3-2: 새로운 코드가 기존의 미디어 쿼리나 반응형 로직과 충돌하지 않도록 검증한다.

## 4. 불필요한 리팩토링 금지
- 4-1: 명확한 기능 개선이나 버그 수정 목적이 아닌 단순한 "코드 정리"를 위해 기존 로직을 대규모로 수정하지 않는다.
- 4-2: 작동 중인 기존 함수의 시그니처나 반환 타입을 동의 없이 변경하지 않는다.

## 5. 디렉토리 구조 및 명명 규칙 준수
- 5-1: `bizinfo-proxy`, `policyfund-api` 등 현재의 폴더 구조를 임의로 재배치하거나 이름을 바꾸지 않는다.
- 5-2: 파일명 및 변수명 생성 시 기존 프로젝트의 네이밍 컨벤션(CamelCase, snake_case 등)을 따른다.

## 6. 외부 라이브러리 도입 최소화
- 6-1: 이미 프로젝트에서 사용 중인 기술 스택(Wrangler, Cloudflare Workers 등)을 최우선으로 활용한다.
- 6-2: 새로운 패키지나 라이브러리 추가가 필요할 경우 반드시 필요성을 먼저 설명하고 승인을 구한다.

## 7. 환경 변수 및 보안 데이터 보호
- 7-1: `.env`, `wrangler.toml` 등의 설정 파일에 포함된 API 키, 시크릿 정보를 로그에 출력하거나 커밋하지 않는다.
- 7-2: 클라이언트 측 코드에 민감한 정보가 노출되지 않도록 엄격히 관리한다.

## 8. 주석 및 문서화 스타일 유지
- 8-1: 기존 코드에 작성된 주석 스타일을 유지하며, 새로운 기능에 대해 명확하고 간결한 주석을 추가한다.
- 8-2: 문서 업데이트 시 기존 내용의 맥락을 해치지 않는다.

## 9. 에셋 및 리소스 경로 보존
- 9-1: `favicon.svg`, 이미지, 아이콘 등 기존 리소스 파일의 경로를 변경하거나 삭제하지 않는다.
- 9-2: 새로운 에셋 추가 시 기존 에셋들과의 시각적 통일성을 고려한다.

## 10. API 엔드포인트 및 데이터 규격 고수
- 10-1: 프록시 서버나 API 서비스의 요청/응답 형식을 임의로 변경하여 프론트엔드와의 연결을 끊지 않는다.
- 10-2: 데이터 스키마 수정 시 하위 호환성을 반드시 고려한다.

## 11. Git 워크플로우 및 커밋 메시지 준수
- 11-1: 사용자의 요청 없이 임의로 푸시하거나 브랜치를 삭제하지 않는다.
- 11-2: 커밋 메시지는 작업 내용을 명확히 알 수 있도록 구체적으로 작성한다.

## 12. SEO 및 메타데이터 무결성
- 12-1: `index.html`의 메타 태그, `robots.txt`, `sitemap.xml` 등 검색 엔진 최적화 설정을 훼손하지 않는다.
- 12-2: 페이지 타이틀이나 설명 문구 변경 시 마케팅적 의도를 사전에 확인한다.

## 13. 성능 및 로딩 속도 저하 방지
- 13-1: 웹 페이지의 로딩 속도를 늦출 수 있는 무거운 스크립트나 최적화되지 않은 이미지를 삽입하지 않는다.
- 13-2: 불필요한 네트워크 요청이 발생하지 않도록 효율적으로 코드를 작성한다.

## 14. 코드 중복 배제 (DRY 원칙)
- 14-1: 기존에 존재하는 유틸리티 함수나 공통 컴포넌트가 있다면 이를 재사용하고, 중복 코드를 생성하지 않는다.
- 14-2: 모듈화된 설계를 존중하며 스파게티 코드를 양산하지 않는다.

## 15. 에러 핸들링 및 예외 처리
- 15-1: 사용자에게 시스템 에러 메시지가 노출되지 않도록 적절한 예외 처리를 수행한다.
- 15-2: 기존의 에러 로깅 패턴이 있다면 이를 따른다.

## 16. 타입 안정성 및 구문 정확성
- 16-1: 문법 오류가 없는지 실행 전 반드시 검토하며, 프로젝트의 타입 시스템을 준수한다.
- 16-2: 런타임 오류를 방지하는 방어적 코딩을 한다.

## 17. 의존성 버전 관리
- 17-1: `package.json` 등 의존성 관리 파일의 버전을 사용자 동의 없이 업그레이드하거나 변경하지 않는다.
- 17-2: 환경의 특수성을 이해하고 작업한다.

## 18. 테스트 및 검증 절차 준수
- 18-1: 변경 사항 적용 후에는 반드시 실제 환경에서의 동작 여부를 확인한다.
- 18-2: 기존 테스트 코드가 있다면 이를 실행하여 회귀 버그를 체크한다.

## 19. 프롬프트 해석의 정확성
- 19-1: 사용자의 요청이 모호할 경우 임의로 판단하지 않고, 반드시 질문을 통해 의도를 확인한다.
- 19-2: 요청받은 범위 밖의 수정을 금지한다.

## 20. 최종 자기 점검 (Self-Audit)
- 20-1: 모든 도구 실행 전, 이 20가지 수칙에 위배되는 사항이 없는지 스스로 검토한다.
- 20-2: 작업 완료 후에는 위 수칙들을 준수했음을 확신해야 한다.
